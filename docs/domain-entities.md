## Перечень сущностей и их атрибутов для веб‑ресурса повышения квалификации

### 1. Назначение документа

Этот документ описывает ключевые бизнес‑сущности системы и их атрибуты на уровне предметной области. Он служит основой для проектирования модели данных (база данных, API) и должен быть понятен как бизнес‑заказчику, так и команде разработки.

Документ опирается на обзорную спецификацию (`overview-spec.md`) и бизнес‑требования (`business-requirements.md`).

**Важно**: в документ не включены сущности, связанные с личным кабинетом слушателя и функционалом преподавателей, так как эти части системы находятся вне рамок MVP.

---

### 2. Группировка сущностей

Сущности сгруппированы по логическим областям:

- **Каталог и расписание**: Направление, Программа, Поток/Набор, Формат обучения
- **Обращения и заявки**: Заявка слушателя, Корпоративный запрос, Запрос звонка/Обращение
- **Контент**: Новость, Кейс, Отзыв
- **Пользователи админ‑панели**: Пользователь административной панели

---

### 3. Каталог и расписание

#### 3.1. Направление (Direction/Category)

**Назначение**: категория/рубрика, к которой относятся программы (например, «Программирование», «Бизнес-анализ», «Финансы»).

**Атрибуты**:
- **id** (уникальный идентификатор) — обязательный
- **название** (текст) — обязательный, уникальный
- **краткое описание** (текст) — необязательный
- **порядок сортировки** (число) — необязательный, для управления порядком отображения на сайте
- **статус** (активен/архив) — обязательный, по умолчанию «активен»
- **дата создания** (дата/время) — обязательный
- **дата обновления** (дата/время) — обязательный

**Связи**:
- Один‑ко‑многим с Программой: одно направление может содержать множество программ

---

#### 3.2. Программа (Program)

**Назначение**: программа повышения квалификации с описанием содержания, форматов, стоимости и других параметров.

**Атрибуты**:
- **id** (уникальный идентификатор) — обязательный
- **название** (текст) — обязательный
- **направление** (ссылка на Направление) — обязательный
- **вид программы** (выбор: "повышение квалификации" или "переподготовка") — обязательный
- **код направления подготовки** (текст) — необязательный
- **лид** (текст) — обязательный, краткий промо‑текст для списков и верхней части карточки (соответствует колонке «Лид» в `pk.csv`)
- **описание «О программе»** (текст/HTML) — обязательный, развернутое описание сути программы (колонка «О ПРОГРАММЕ»)
- **программа обучения (учебный план)** (текст/HTML) — обязательный, структурированное описание модулей и тем (колонка «Программа обучения (Учебный план)»)
- **целевая аудитория** (текст) — обязательный, для кого предназначена программа (колонка «Для кого предназначена программа»)
- **порядок зачисления** (текст) — необязательный, описание шага записи/зачисления на программу (колонка «Порядок зачисления на программу»)
- **форматы обучения** (множественный выбор из Форматов обучения) — обязательный, минимум один формат (колонка «Форма обучения»)
- **объём программы в часах** (число) — обязательный, учебная трудоёмкость в академических часах (колонка «Объем учебной программы (часы)/ трудоёмкость»)
- **календарная длительность** (текст) — обязательный, например «3 дня», «1 месяц 1 неделя» (колонка «Длительность»)
- **стоимость** (число или текст) — обязательный, может быть диапазоном или «по запросу» (колонка «Стоимость»)
- **итог по программе** (текст) — необязательный, что получает слушатель по завершении (удостоверение, лицензия и т.п.; колонка «Итог»)
- **требования к слушателям** (текст) — необязательный, предварительные знания/навыки
- **результаты обучения** (текст/список) — необязательный, что получит слушатель по итогам
- **статус** (активен/архив/черновик) — обязательный, по умолчанию «черновик»
- **дата создания** (дата/время) — обязательный
- **дата обновления** (дата/время) — обязательный

**Связи**:
- Многие‑к‑одному с Направлением: программа принадлежит одному направлению
- Один‑ко‑многим с Потоком: одна программа может иметь множество потоков/наборов
- Многие‑ко‑многим с Форматом обучения: программа может проводиться в нескольких форматах

---

#### 3.3. Поток/Набор (CourseBatch/Stream)

**Назначение**: конкретный запуск программы с датами начала/окончания, форматом проведения и расписанием занятий.

**Атрибуты**:
- **id** (уникальный идентификатор) — обязательный
- **программа** (ссылка на Программу) — обязательный
- **название/номер потока** (текст) — необязательный, для идентификации (например, «Поток 2024-01»)
- **дата начала** (дата) — обязательный
- **дата окончания** (дата) — необязательный
- **формат проведения** (ссылка на Формат обучения) — обязательный
- **расписание** (текст) — необязательный, краткое описание расписания (например, «Пн, Ср, Пт с 18:00 до 21:00»)
- **количество мест** (число) — необязательный
- **стоимость** (текст) — необязательный, стоимость потока (например: "15000 руб", "Бесплатно"), по умолчанию "0"
- **статус** (набор открыт/набор закрыт/идет обучение/завершен/отменен) — обязательный, по умолчанию «набор открыт»
- **дата создания** (дата/время) — обязательный
- **дата обновления** (дата/время) — обязательный

**Связи**:
- Многие‑к‑одному с Программой: поток относится к одной программе
- Многие‑к‑одному с Форматом обучения: поток проводится в одном формате

---

#### 3.4. Формат обучения (LearningFormat)

**Назначение**: способ проведения обучения (очный, онлайн, смешанный и т.п.).

**Атрибуты**:
- **id** (уникальный идентификатор) — обязательный
- **название** (текст) — обязательный, уникальный (например, «Очный», «Онлайн с преподавателем», «Заочный»)
- **краткое описание** (текст) — необязательный
- **подробное описание** (текст/HTML) — необязательный, для страницы «Форматы обучения»
- **порядок сортировки** (число) — необязательный
- **статус** (активен/архив) — обязательный, по умолчанию «активен»
- **дата создания** (дата/время) — обязательный
- **дата обновления** (дата/время) — обязательный

**Связи**:
- Многие‑ко‑многим с Программой: формат может использоваться в разных программах, программа может проводиться в разных форматах
- Один‑ко‑многим с Потоком: поток проводится в одном формате

---

### 4. Обращения и заявки

#### 4.1. Заявка слушателя (Application)

**Назначение**: заявка физического лица на обучение по одной или нескольким программам.

**Атрибуты**:
- **id** (уникальный идентификатор) — обязательный
- **программы** (множественные ссылки на Программы) — обязательный, минимум одна программа
- **поток** (ссылка на Поток) — необязательный, если пользователь выбрал конкретный поток
- **ФИО слушателя** (текст) — обязательный
- **телефон** (текст) — обязательный
- **email** (текст) — необязательный, но рекомендуется
- **предпочтительный способ связи** (выбор: телефон/email/мессенджер) — необязательный
- **комментарий** (текст) — необязательный, дополнительная информация от слушателя
- **статус** (новая/в работе/подтверждена/отклонена/завершена) — обязательный, по умолчанию «новая»
- **дата создания** (дата/время) — обязательный
- **дата обновления** (дата/время) — обязательный
- **комментарий администратора** (текст) — необязательный, внутренние заметки

**Связи**:
- Многие‑ко‑многим с Программой: заявка может относиться к нескольким программам
- Многие‑к‑одному с Потоком: заявка может быть привязана к конкретному потоку (необязательно)

---

#### 4.2. Корпоративный запрос (CorporateRequest)

**Назначение**: запрос организации на коммерческое предложение по корпоративному обучению.

**Атрибуты**:
- **id** (уникальный идентификатор) — обязательный
- **название организации** (текст) — обязательный
- **ФИО контактного лица** (текст) — обязательный
- **должность контактного лица** (текст) — необязательный
- **телефон** (текст) — обязательный
- **email** (текст) — обязательный
- **интересующие направления/темы** (текст) — обязательный, может быть свободный текст или ссылки на Направления
- **интересующие программы** (текст или множественные ссылки на Программы) — необязательный
- **ориентировочное количество сотрудников** (число) — обязательный
- **желаемые сроки** (текст) — необязательный
- **дополнительные комментарии** (текст) — необязательный
- **статус** (новый/в работе/КП отправлено/закрыт) — обязательный, по умолчанию «новый»
- **дата создания** (дата/время) — обязательный
- **дата обновления** (дата/время) — обязательный
- **комментарий администратора** (текст) — необязательный, внутренние заметки

**Связи**:
- Многие‑ко‑многим с Направлением: запрос может относиться к нескольким направлениям (необязательно, если используется текстовое поле)
- Многие‑ко‑многим с Программой: запрос может относиться к конкретным программам (необязательно)

---

#### 4.3. Запрос звонка/Обращение (CallbackRequest)

**Назначение**: краткое обращение пользователя (физ. лица или организации) с просьбой связаться.

*Для MVP реализована упрощённая версия с полями: имя, телефон, email, дата создания.*

**Атрибуты**:
- **id** (уникальный идентификатор) — обязательный
- **имя** (текст) — обязательный
- **телефон** (текст) — обязательный
- **email** (текст) — необязательный
- **тип обращения** (выбор: физ. лицо/организация) — необязательный
- **комментарий/вопрос** (текст) — необязательный
- **статус** (новый/обработан/закрыт) — обязательный, по умолчанию «новый»
- **дата создания** (дата/время) — обязательный
- **дата обновления** (дата/время) — обязательный
- **комментарий администратора** (текст) — необязательный

**Связи**: нет прямых связей с другими сущностями (самостоятельное обращение)

---

### 5. Контент

#### 5.1. Новость (News)

**Назначение**: новость или статья для публикации на сайте.

**Атрибуты**:
- **id** (уникальный идентификатор) — обязательный
- **заголовок** (текст) — обязательный
- **краткое описание/анонс** (текст) — необязательный, для списков новостей
- **полный текст** (текст/HTML) — обязательный
- **дата публикации** (дата) — обязательный
- **изображение/превью** (ссылка на файл или URL) — необязательный
- **категория** (текст или ссылка на категорию) — необязательный
- **статус** (черновик/опубликовано/архив) — обязательный, по умолчанию «черновик»
- **дата создания** (дата/время) — обязательный
- **дата обновления** (дата/время) — обязательный

**Связи**: нет обязательных связей с другими сущностями

---

#### 5.2. Кейс (Case)

**Назначение**: описание реализованного проекта корпоративного обучения для демонстрации опыта института.

**Атрибуты**:
- **id** (уникальный идентификатор) — обязательный
- **название компании** (текст) — обязательный
- **логотип компании** (ссылка на файл или URL) — необязательный
- **заголовок кейса** (текст) — обязательный
- **задача** (текст/HTML) — обязательный, описание задачи клиента
- **подход/решение** (текст/HTML) — необязательный, как институт решил задачу
- **результат** (текст/HTML) — обязательный, что получилось в итоге
- **направления** (множественные ссылки на Направления или текст) — необязательный
- **программы** (множественные ссылки на Программы) — необязательный
- **статус** (черновик/опубликовано/архив) — обязательный, по умолчанию «черновик»
- **дата создания** (дата/время) — обязательный
- **дата обновления** (дата/время) — обязательный

**Связи**:
- Многие‑ко‑многим с Направлением: кейс может относиться к нескольким направлениям (необязательно)
- Многие‑ко‑многим с Программой: кейс может упоминать конкретные программы (необязательно)

---

#### 5.3. Отзыв (Review)

**Назначение**: отзыв слушателя или компании о программах/обучении в институте.

**Атрибуты**:
- **id** (уникальный идентификатор) — обязательный
- **тип отзыва** (выбор: от слушателя/от компании) — обязательный
- **ФИО автора** (текст) — обязательный
- **должность** (текст) — необязательный, для отзывов от компаний
- **название компании** (текст) — необязательный, для корпоративных отзывов
- **логотип компании** (ссылка на файл или URL) — необязательный
- **текст отзыва** (текст/HTML) — обязательный
- **программа** (ссылка на Программу) — необязательный, если отзыв относится к конкретной программе
- **рейтинг/оценка** (число от 1 до 5) — необязательный
- **фото автора** (ссылка на файл или URL) — необязательный
- **статус** (черновик/опубликовано/скрыто) — обязательный, по умолчанию «черновик»
- **дата создания** (дата/время) — обязательный
- **дата обновления** (дата/время) — обязательный

**Связи**:
- Многие‑к‑одному с Программой: отзыв может относиться к конкретной программе (необязательно)

---

### 6. Пользователи админ‑панели

#### 6.1. Пользователь административной панели (AdminUser)

**Назначение**: сотрудник института, имеющий доступ к административной панели для управления контентом, программами и заявками.

**Атрибуты**:
- **id** (уникальный идентификатор) — обязательный
- **логин/email** (текст) — обязательный, уникальный
- **ФИО** (текст) — обязательный
- **роль** (выбор: администратор/методист/редактор) — обязательный, определяет уровень доступа
- **статус** (активен/заблокирован) — обязательный, по умолчанию «активен»
- **дата создания** (дата/время) — обязательный
- **дата последнего входа** (дата/время) — необязательный

**Примечание**: аутентификация и авторизация (пароли, токены и т.п.) выносятся на уровень технической реализации и в данном документе не детализируются.

**Связи**: нет прямых связей с другими бизнес‑сущностями (пользователь работает с ними через интерфейс админки)

### 8. Дополнительные замечания

- **Идентификаторы**: все сущности должны иметь уникальный идентификатор (id), тип которого определяется на этапе технического проектирования.

- **Даты создания и обновления**: рекомендуется хранить для всех сущностей для аудита и отслеживания изменений.

- **Статусы**: большинство сущностей имеют поле «статус» для управления жизненным циклом (публикация, архивация и т.п.). Конкретные значения статусов могут быть уточнены на этапе разработки.

- **Мягкое удаление**: рекомендуется использовать статусы «архив»/«скрыто» вместо физического удаления записей для сохранения истории.

- **Валидация**: требования к формату и обязательности полей описаны на бизнес‑уровне. Технические детали валидации (регулярные выражения, ограничения длины и т.п.) определяются на этапе разработки.

- **Медиафайлы**: изображения, логотипы и другие файлы могут храниться как ссылки на файловое хранилище или URL. Конкретный механизм хранения определяется архитектурно.

### 9. Использование документа

Данный документ служит основой для:
- проектирования схемы базы данных;
- проектирования API (структуры запросов/ответов);
- написания технических спецификаций;
- планирования разработки модулей системы.

При необходимости документ может быть дополнен:

- диаграммами ER (entity-relationship);
- описанием бизнес‑правил и валидаций более детально.
