## Типы пользователей и их use cases для веб‑ресурса повышения квалификации

### 1. Цель документа

Этот документ описывает основные типы пользователей веб‑ресурса повышения квалификации института и их ключевые use cases.  
Он опирается на документ «Бизнес-спецификация: образ и границы веб‑ресурса повышения квалификации» и служит основой для дальнейшей формализации бизнес‑требований и пользовательских историй.

Типы пользователей и сценарии во многом вдохновлены практикой Бауманского учебного центра «Специалист» ([specialist.ru](https://www.specialist.ru/)), адаптированной под контекст института.

### 2. Типы пользователей (роли)

1. **Слушатель (физическое лицо)**  
   Частное лицо, которое рассматривает программы повышения квалификации для себя.

2. **Представитель организации (корпоративный клиент)**  
   HR‑специалист, руководитель подразделения или сотрудник, отвечающий за обучение персонала.

3. **Администратор / методист**  
   Сотрудник института, который управляет каталогом курсов, расписанием, заявками и контентом.

Ниже для каждой роли описаны цели, контекст использования и ключевые use cases.

---

### 3. Слушатель (физическое лицо)

#### 3.1. Цели и контекст

- Найти подходящую программу повышения квалификации.
- Понять формат, стоимость, длительность и требования.
- Удобно подать заявку и отслеживать её статус.

Контекст:  
— Использует сайт эпизодически (несколько раз до старта курса и при необходимости уточнения информации).  
— Может заходить как с рабочего ПК, так и с мобильных устройств.

#### 3.2. Use case 1: Поиск курса по направлению и интересам

- **Триггер**: пользователь понимает, что ему нужно повысить квалификацию в определённой области (например, «Python», «Excel для аналитики» и т.п.).
- **Цель**: найти список релевантных курсов и сузить выбор до 1–2 вариантов.
- **Основной поток**:
  1. Пользователь открывает главную страницу.
  2. Переходит к списку направлений или использует поиск.
  3. Выбирает направление (например, «Программирование», «Бизнес-анализ»).
  4. Фильтрует курсы по уровню, формату, датам или просто просматривает список.
  5. Открывает карточки интересующих курсов.
- **Результат**: пользователь получает короткий список курсов, которые потенциально ему подходят.

#### 3.3. Use case 2: Изучение подробной информации о курсе

- **Триггер**: пользователь выбрал конкретный курс в каталоге.
- **Цель**: понять, подходит ли курс по содержанию, уровню и формату.
- **Основной поток**:
  1. Пользователь переходит в карточку курса.
  2. Читает краткое описание и результаты обучения.
  3. Просматривает подробную программу по разделам/модулям.
  4. Изучает информацию о длительности, формате (очный/онлайн/смешанный), расписании и стоимости.
  5. При необходимости сравнивает этот курс с другими.
- **Результат**: пользователь принимает решение «подходит / не подходит» и либо уходит, либо оформляет заявку.

#### 3.4. Use case 3: Оформление заявки на курс

- **Триггер**: пользователь решил записаться на конкретный курс.
- **Цель**: отправить корректно заполненную заявку, чтобы с ним связались или подтвердили участие.
- **Основной поток**:
  1. На странице курса пользователь нажимает «Оставить заявку».
  2. Заполняет форму (ФИО, контакты, выбранную дату/поток, предпочитаемый формат связи).
  3. Подтверждает согласие на обработку персональных данных.
  4. Отправляет заявку.
  5. Получает экран/сообщение с подтверждением отправки и базовой информацией о дальнейших шагах.
- **Результат**: заявка зарегистрирована в системе, пользователь информирован о её отправке.

### 4. Представитель организации (корпоративный клиент)

#### 4.1. Цели и контекст

- Оценить возможности института по корпоративному обучению.
- Подобрать программы для группы сотрудников под задачи компании.
- Получить коммерческое предложение и согласовать условия.

Контекст:  
— Использует сайт эпизодически, но при каждом заходе решает конкретную задачу (подготовка КП, выбор поставщика обучения).  
— Может заходить с рабочего ПК, часто из корпоративной сети.

#### 4.2. Use case 1: Ознакомление с предложением для корпоративных клиентов

- **Триггер**: компания рассматривает институт как потенциального поставщика обучения.
- **Цель**: понять формат сотрудничества, направления и опыт работы с корпоративными заказчиками.
- **Основной поток**:
  1. Пользователь переходит в раздел «Для организаций».
  2. Изучает описание услуг, форматов и преимуществ.
  3. Просматривает список направлений и типов программ для бизнеса.
  4. При необходимости переходит к кейсам и отзывам компаний.
- **Результат**: пользователь формирует представление о возможностях института.

#### 4.3. Use case 2: Просмотр кейсов и отзывов корпоративных клиентов

- **Триггер**: пользователь хочет оценить опыт института в похожих проектах.
- **Цель**: увидеть реальные кейсы, примеры внедрений и результаты.
- **Основной поток**:
  1. Переходит в раздел кейсов/отзывов.
  2. Выбирает интересующие отрасли или типы проектов (при наличии фильтрации).
  3. Открывает карточки кейсов.
  4. Просматривает задачу, подход и результат.
- **Результат**: повышается доверие и понимание компетенций института.

#### 4.4. Use case 3: Запрос коммерческого предложения

- **Триггер**: компания заинтересована в конкретном обучающем решении.
- **Цель**: отправить запрос, достаточный для подготовки КП.
- **Основной поток**:
  1. Пользователь открывает форму «Запрос КП» или «Корпоративное обучение для вашей команды».
  2. Заполняет данные об организации, контактном лице, интересующих направлениях/курсах, объёме (кол-во сотрудников) и желаемых сроках.
  3. Подтверждает согласие на обработку персональных данных.
  4. Отправляет форму.
  5. Получает подтверждение, что менеджер свяжется в рабочее время.
- **Результат**: запрос успешно создан в системе и передан ответственным менеджерам.

#### 4.5. Use case 4: Обратная связь / заказ звонка

- **Триггер**: у пользователя остаются вопросы, которые проще обсудить голосом.
- **Цель**: оставить контакты для обратного звонка.
- **Основной поток**:
  1. Пользователь открывает форму «Заказать звонок» или всплывающее окно обратной связи.
  2. Заполняет имя, телефон и краткий комментарий.
  3. Подтверждает согласие на обработку персональных данных.
  4. Отправляет форму.
- **Результат**: запрос на звонок зафиксирован; менеджер получает задачу связаться с клиентом.

### 5. Администратор / методист

#### 5.1. Цели и контекст

- Поддерживать актуальный каталог курсов и направлений.
- Управлять расписанием и наборами групп.
- Работать с заявками (физических лиц и корпоративных клиентов).
- Публиковать новости, кейсы и отзывы.

Контекст:  
— Работает с системой регулярно, как с внутренним инструментом.  
— Заходит, как правило, с рабочего ПК через защищённое соединение.

#### 5.2. Use case 1: Создание и редактирование курсов

- **Триггер**: нужно добавить новый курс или скорректировать существующий.
- **Цель**: обеспечить наличие в каталоге актуальной информации о программах.
- **Основной поток**:
  1. Администратор входит в административную панель.
  2. Переходит в раздел управления курсами.
  3. Создаёт новый курс или редактирует существующий.
  4. Заполняет/обновляет ключевые поля (название, описание, программа, форматы, стоимость).
  5. Сохраняет изменения и при необходимости публикует курс.
- **Результат**: курс появляется или обновляется в публичном каталоге.

#### 5.3. Use case 2: Управление расписанием и наборами

- **Триггер**: запуск нового потока курса или изменение расписания.
- **Цель**: предоставить слушателям актуальную информацию о датах и расписании.
- **Основной поток**:
  1. Администратор открывает страницу курса/потоков.
  2. Добавляет новый поток (даты начала/окончания, формат, расписание) или редактирует существующий.
  3. Сохраняет изменения.
- **Результат**: расписание на сайте обновлено и доступно пользователям.

#### 5.4. Use case 3: Обработка заявок слушателей

- **Триггер**: поступают новые заявки на курсы от физических лиц.
- **Цель**: проверить заявки, присвоить статусы, передать в дальнейшую обработку.
- **Основной поток**:
  1. Администратор открывает раздел заявок.
  2. Фильтрует заявки по курсам, статусам, датам.
  3. Открывает карточку заявки, проверяет корректность данных.
  4. Изменяет статус (например, «Новая» → «В работе» → «Подтверждена»).
  5. При необходимости экспортирует данные или передаёт их во внутренние системы.
- **Результат**: заявки обработаны, их статусы отражают реальное состояние.

#### 5.5. Use case 4: Обработка корпоративных запросов

- **Триггер**: поступают запросы на коммерческие предложения от организаций.
- **Цель**: системно учитывать и обрабатывать запросы корпоративных клиентов.
- **Основной поток**:
  1. Администратор открывает список корпоративных заявок.
  2. Просматривает детали каждой заявки (организация, запрос, контакты).
  3. Назначает ответственного менеджера (при наличии такого механизма) или фиксирует комментарий.
  4. Обновляет статус запроса.
- **Результат**: корпоративные запросы упорядочены и сопровождаются по понятному процессу.

#### 5.6. Use case 5: Публикация новостей, кейсов и отзывов

- **Триггер**: необходимо добавить новый материал (новость, статью, кейс, отзыв).
- **Цель**: поддерживать актуальность и привлекательность публичной части сайта.
- **Основной поток**:
  1. Администратор заходит в раздел «Контент» админ‑панели.
  2. Выбирает тип материала (новость, кейс, отзыв).
  3. Заполняет заголовок, текст, иллюстрации и вспомогательные поля (дата, категория и т.п.).
  4. Сохраняет и публикует материал.
- **Результат**: новый материал появляется в соответствующих разделах сайта.

### 7. Использование документа дальше

- На основе описанных типов пользователей и их use cases можно:
  - формализовать **бизнес‑требования** к функциональным блокам системы;
  - развернуть **user stories** для продукта и команды разработки;
  - уточнить **приоритеты реализации** MVP и следующих итераций.
- Документ должен рассматриваться вместе с общей бизнес‑спецификацией (образ и границы продукта), чтобы обеспечить целостное видение системы.
